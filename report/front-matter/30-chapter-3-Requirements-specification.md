# Capítulo III: Requirements Specification. 
## 3.1. User Stories

| Epic / Story ID | Título | Descripción | Criterios de Aceptación | Relacionado con (Epic ID) |
|---|---|---|---|---|
| **EP01** | **Gestión de Inventario** | Los dueños y encargados de restaurantes chifa necesitan controlar sus insumos de forma centralizada, registrando entradas, salidas y niveles de stock para evitar quiebres o excesos. | | |
| **EP02** | **Abastecimiento y Órdenes de Compra** | Los restaurantes y proveedores necesitan coordinar pedidos de insumos de forma digital, reemplazando llamadas y mensajes informales por un flujo estructurado dentro de la plataforma. | | |
| **EP03** | **Monitoreo Operativo y Alertas IoT** | Los administradores de restaurantes necesitan supervisar condiciones críticas de cocina y almacén (temperatura, humedad) mediante sensores para prevenir riesgos y pérdidas. | | |
| **EP04** | **Panel del Proveedor** | Los proveedores necesitan visibilidad sobre la demanda de sus clientes, gestionar su catálogo y hacer seguimiento de pedidos para planificar mejor su distribución. | | |
| **EP05** | **Plataforma y Acceso** | Todos los usuarios necesitan acceder de forma segura a la plataforma, gestionar sus cuentas, planes de suscripción y recibir soporte, garantizando una experiencia fluida desde cualquier dispositivo. | | |
| **EP06** | **Comandas y Órdenes para Cocina** | Los restaurantes necesitan gestionar las comandas de manera eficiente, desde la toma del pedido hasta la preparación en cocina, garantizando un servicio rápido y sin errores. | | |
| **EP07** | **Landing Page** | Los potenciales clientes (dueños de restaurantes y proveedores) necesitan una página web atractiva e informativa que presente las características de SupplyWok, planes disponibles y permita el contacto directo para conocer más sobre la plataforma. | | |
| US01 | Registro de inventario inicial | **Como** dueño de un restaurante chifa, **quiero** registrar todos mis insumos actuales con cantidades y unidades, **para** tener una base de datos centralizada que reemplace el control manual. | **Escenario 1:** **Dado que** el dueño accede al módulo de inventario. **Cuando** completa el formulario con nombre, categoría, cantidad y unidad de un insumo. **Entonces** el insumo queda guardado y visible en el listado de inventario. <br> <br> **Escenario 2:** **Dado que** el dueño desea corregir un registro. **Cuando** edita o elimina un insumo existente. **Entonces** los cambios se reflejan de inmediato en el inventario. | EP01 |
| US02 | Actualización de stock por consumo diario | **Como** encargado de almacén, **quiero** registrar el consumo diario de insumos, **para** mantener el inventario actualizado sin depender de conteos manuales al cierre del día. | **Escenario 1:** **Dado que** el encargado accede al módulo de movimientos. **Cuando** registra una salida con insumo, cantidad y fecha. **Entonces** el stock disponible se descuenta automáticamente. <br> <br> **Escenario 2:** **Dado que** el encargado desea revisar movimientos anteriores. **Cuando** consulta el historial de un insumo. **Entonces** se muestran todas las salidas registradas con fecha y cantidad. | EP01 |
| US03 | Configuración de alertas de stock mínimo | **Como** administrador de un restaurante chifa, **quiero** definir un nivel mínimo de stock por insumo, **para** recibir una alerta cuando el inventario esté próximo a agotarse. | **Escenario 1:** **Dado que** el administrador configura el umbral de un insumo. **Cuando** el stock cae por debajo del valor definido. **Entonces** se muestra una alerta visible en el dashboard. <br> <br> **Escenario 2:** **Dado que** el administrador desea ser notificado fuera del sistema. **Cuando** activa la opción de alertas por correo. **Entonces** recibe un correo al momento en que se activa la alerta. | EP01 |
| US04 | Visualización del dashboard de inventario | **Como** dueño de un restaurante chifa, **quiero** ver un panel visual con el estado de todos mis insumos, **para** identificar rápidamente cuáles están en nivel crítico, normal o con exceso. | **Escenario 1:** **Dado que** el dueño accede al dashboard. **Cuando** se carga el panel principal. **Entonces** los insumos se muestran clasificados por estado (crítico, normal, exceso) con indicadores de color. <br> <br> **Escenario 2:** **Dado que** se registra un movimiento de stock. **Cuando** el cambio afecta el nivel de un insumo. **Entonces** el dashboard se actualiza en tiempo real sin recargar la página. | EP01 |
| US05 | Registro de entrada de mercadería | **Como** encargado de almacén, **quiero** registrar la llegada de insumos con sus cantidades y proveedor de origen, **para** actualizar el stock disponible y mantener el historial de entradas. | **Escenario 1:** **Dado que** el encargado accede al módulo de entradas. **Cuando** registra una entrada con insumo, cantidad, proveedor y fecha. **Entonces** el stock del insumo se incrementa con la cantidad recibida y el registro queda guardado en el historial de entradas. <br> <br> **Escenario 2:** **Dado que** el encargado desea revisar entradas anteriores. **Cuando** consulta el historial de entradas de un insumo. **Entonces** se muestran todos los registros con fecha, proveedor y cantidad recibida. <br> <br> **Escenario 3:** **Dado que** el encargado intenta guardar una entrada. **Cuando** deja campos obligatorios como insumo o cantidad vacíos. **Entonces** el sistema muestra un mensaje de error indicando los campos requeridos y no guarda el registro. | EP01 |
| US06 | Exportar reportes en PDF/CSV | **Como** dueño del restaurante, **quiero** poder exportar archivos en formato PDF o CSV, **para** poder usarlos en el manejo del negocio. | **Escenario 1:** **Dado que** el encargado quiere obtener reportes para almacenar localmente. **Cuando** interactue con el boton correspondiente y el sistema no detecte errores. **Entonces** se genera el reporte en formato PDF o CSV para su uso. <br> <br> **Escenario 2:** **Dado que** el encargado quiere poder imprimir el reporte. **Cuando** exporte el reporte en los formatos permitidos. **Entonces** se guarda el reporte en el dispositivo del usuario. | EP01 |
| US07 | Registro de baja de insumos | **Como** encargado de almacén, **quiero** registrar insumos dados de baja indicando el motivo, **para** descontarlos del inventario y mantener un historial de pérdidas. | **Escenario 1:** **Dado que** el encargado accede al módulo de bajas. **Cuando** registra una baja con insumo, cantidad y motivo (vencimiento, daño u otro). **Entonces** el stock del insumo se descuenta y el registro queda en el historial de bajas con fecha y motivo. <br> <br> **Escenario 2:** **Dado que** el encargado desea revisar bajas anteriores de un insumo. **Cuando** consulta su historial. **Entonces** se muestran las bajas registradas con fecha, motivo y cantidad. <br> <br> **Escenario 3:** **Dado que** el encargado intenta registrar una baja. **Cuando** la cantidad ingresada supera el stock disponible del insumo. **Entonces** el sistema muestra un mensaje de error indicando que no hay suficiente stock para dar de baja y no realiza el descuento. | EP01 |
| US08 | Visualización de insumos que requieren reabastecimiento | **Como** dueño de un restaurante chifa, **quiero** ver una lista de insumos que han alcanzado o caído por debajo de su stock mínimo configurado, **para** identificar rápidamente qué productos necesito pedir. | **Escenario 1:** **Dado que** el dueño accede a la sección de abastecimiento. **Cuando** al menos un insumo tiene stock igual o menor al mínimo configurado. **Entonces** el sistema muestra una lista con esos insumos, su stock actual y su stock mínimo. <br> <br> **Escenario 2:** **Dado que** el dueño revisa la lista de insumos por reabastecer. **Cuando** selecciona un insumo de la lista. **Entonces** puede iniciar la creación de una orden de compra con ese insumo precompletado. <br> <br> **Escenario 3:** **Dado que** el dueño accede a la sección de abastecimiento. **Cuando** ningún insumo ha alcanzado su stock mínimo. **Entonces** el sistema muestra un mensaje indicando que todos los insumos están dentro de los niveles aceptables. | EP02 |
| US09 | Creación de orden de compra | **Como** administrador de un restaurante chifa, **quiero** generar una orden de compra seleccionando insumos, cantidades y proveedor, **para** centralizar y registrar mis solicitudes de abastecimiento en la plataforma. | **Escenario 1:** **Dado que** el administrador accede al módulo de pedidos. **Cuando** completa el formulario con al menos un insumo, su cantidad y el proveedor destino. **Entonces** la orden queda registrada con estado "Pendiente" y aparece en el historial de pedidos. <br> <br> **Escenario 2:** **Dado que** el administrador visualiza una orden con estado "Pendiente". **Cuando** decide modificarla antes de que el proveedor la confirme. **Entonces** puede editar los insumos, cantidades o cancelar la orden. <br> <br> **Escenario 3:** **Dado que** el administrador intenta crear una orden. **Cuando** no selecciona ningún insumo o no elige un proveedor destino. **Entonces** el sistema muestra un mensaje de error indicando los campos obligatorios y no registra la orden. | EP02 | 
| US10 | Historial de órdenes de compra | **Como** dueño de un restaurante chifa, **quiero** acceder al historial completo de mis órdenes de compra anteriores, **para** analizar patrones de abastecimiento y detectar ineficiencias. | **Escenario 1:** **Dado que** el dueño accede al módulo de historial. **Cuando** se carga el listado. **Entonces** se muestran todas las órdenes con fecha, proveedor, insumos, cantidades y estado. <br> <br> **Escenario 2:** **Dado que** el dueño desea filtrar resultados. **Cuando** aplica filtros por proveedor, fecha o estado. **Entonces** el listado se actualiza mostrando solo las órdenes que coinciden. <br> <br> **Escenario 3:** **Dado que** el dueño accede al módulo de historial. **Cuando** no existe ninguna orden de compra registrada. **Entonces** el sistema muestra un mensaje informativo indicando que aún no hay órdenes registradas. | EP02 | 
| US11 | Proyección de demanda basada en historial | **Como** administrador de un restaurante chifa, **quiero** visualizar una proyección de consumo de insumos para los próximos 7 días, **para** anticipar compras antes de que se agoten los productos. | **Escenario 1:** **Dado que** el administrador accede al módulo de proyección. **Cuando** se carga el panel. **Entonces** el sistema muestra el consumo proyectado por insumo para los próximos 7 días basado en el historial. <br> <br> **Escenario 2:** **Dado que** el administrador desea ver la proyección en formato visual. **Cuando** accede al gráfico de proyección. **Entonces** se presenta una línea de tendencia por insumo seleccionado. <br> <br> **Escenario 3:** **Dado que** el administrador accede al módulo de proyección. **Cuando** el historial de consumo disponible es insuficiente para calcular una tendencia. **Entonces** el sistema muestra un aviso indicando que se necesita más historial registrado para proyectar la demanda. | EP02 | 
| US12 | Gestión de proveedores vinculados | **Como** dueño de un restaurante chifa, **quiero** registrar y gestionar la lista de proveedores con los que trabajo, **para** tener un directorio centralizado y asociar cada insumo a su proveedor. | **Escenario 1:** **Dado que** el dueño accede al módulo de proveedores. **Cuando** registra un nuevo proveedor con nombre, contacto y categoría de insumos. **Entonces** el proveedor queda disponible para ser asociado a órdenes e insumos. <br> <br> **Escenario 2:** **Dado que** el dueño selecciona un insumo del inventario. **Cuando** revisa su detalle. **Entonces** puede ver qué proveedor lo surte y acceder a su perfil directamente. <br> <br> **Escenario 3:** **Dado que** el dueño intenta registrar un nuevo proveedor. **Cuando** deja campos obligatorios vacíos como nombre o contacto. **Entonces** el sistema muestra un mensaje de error señalando los campos requeridos y no guarda el registro. | EP02 | 
| US13 | Seguimiento de estado de órdenes activas | **Como** administrador de un restaurante chifa, **quiero** hacer seguimiento en tiempo real del estado de mis órdenes de compra activas, **para** saber si un pedido está pendiente, en preparación o ya fue despachado. | **Escenario 1:** **Dado que** el administrador accede al panel de órdenes activas. **Cuando** se carga la vista. **Entonces** se muestran todas las órdenes en curso con su estado actualizado. <br> <br> **Escenario 2:** **Dado que** el proveedor actualiza el estado de una orden. **Cuando** el cambio se registra en el sistema. **Entonces** el restaurante recibe una notificación y el estado en su panel se actualiza automáticamente. <br> <br> **Escenario 3:** **Dado que** el administrador accede al panel de órdenes activas. **Cuando** no existen órdenes en curso. **Entonces** el sistema muestra un mensaje informativo indicando que no hay pedidos activos en este momento. | EP02 | 
| US14 | Monitoreo de temperatura en almacén | **Como** dueño de un restaurante chifa, **quiero** visualizar la temperatura actual de mi almacén mediante sensores IoT, **para** detectar condiciones que puedan afectar la calidad de los insumos. | **Escenario 1:** **Dado que** el sensor IoT está activo y conectado. **Cuando** el dueño accede al panel de monitoreo. **Entonces** se muestra la temperatura actual y un historial de las últimas horas. <br> <br> **Escenario 2:** **Dado que** la temperatura supera el rango configurado. **Cuando** el sensor detecta la condición. **Entonces** el sistema genera una alerta inmediata visible en el dashboard y registrada con fecha y hora. <br> <br> **Escenario 3:** **Dado que** el dueño accede al panel de monitoreo. **Cuando** el sensor IoT no está conectado o no envía datos. **Entonces** el sistema muestra un aviso indicando que el sensor está desconectado y no puede mostrar datos en tiempo real. | EP03 |
| US15 | Alertas de riesgo en cocina | **Como** administrador de un restaurante chifa, **quiero** recibir alertas cuando se detecten condiciones de riesgo en cocina (temperatura, humedad u otros parámetros), **para** actuar de forma preventiva y proteger al personal. | **Escenario 1:** **Dado que** los sensores IoT de cocina están activos. **Cuando** un parámetro supera el rango seguro configurado. **Entonces** se genera una alerta visible en el dashboard y se registra en el log de incidencias. <br> <br> **Escenario 2:** **Dado que** el administrador revisa el log. **Cuando** accede al historial de alertas. **Entonces** puede ver fecha, hora, tipo de alerta y parámetro que la generó. <br> <br> **Escenario 3:** **Dado que** el sistema intenta generar una alerta. **Cuando** no hay rangos seguros configurados para la variable que superó el umbral. **Entonces** el sistema no genera la alerta y muestra un aviso indicando que se deben configurar los rangos primero. | EP03 | 
| US16 | Configuración de rangos seguros por variable | **Como** administrador de un restaurante chifa, **quiero** configurar los rangos aceptables de temperatura y humedad para cocina y almacén, **para** que las alertas se activen según los parámetros de mi operación. | **Escenario 1:** **Dado que** el administrador accede a la configuración de monitoreo. **Cuando** define los valores mínimo y máximo para una variable (temperatura, humedad). **Entonces** el sistema usa esos valores como umbral para generar alertas. <br> <br> **Escenario 2:** **Dado que** el administrador desea ajustar un rango existente. **Cuando** edita los valores y guarda los cambios. **Entonces** las alertas futuras se generan con los nuevos parámetros configurados. <br> <br> **Escenario 3:** **Dado que** el administrador intenta guardar un rango. **Cuando** el valor mínimo ingresado es mayor o igual al valor máximo. **Entonces** el sistema muestra un mensaje de error indicando que el rango es inválido y no guarda los cambios. | EP03 | 
| US17 | Control de ocupación de mesas | **Como** administrador de un restaurante chifa, **quiero** visualizar el estado de ocupación de las mesas en tiempo real, **para** coordinar el flujo del servicio y anticipar la demanda de insumos en horas pico. | **Escenario 1:** **Dado que** el administrador accede al módulo de salón. **Cuando** se carga el panel. **Entonces** se muestra un mapa visual de las mesas con su estado actual (libre, ocupada, en espera). <br> <br> **Escenario 2:** **Dado que** el personal actualiza el estado de una mesa. **Cuando** el cambio se registra. **Entonces** el mapa se actualiza en tiempo real sin necesidad de recargar la página. <br> <br> **Escenario 3:** **Dado que** el administrador accede al módulo de salón. **Cuando** no hay mesas registradas en el sistema. **Entonces** el sistema muestra un mensaje indicando que se deben registrar mesas antes de visualizar el panel de ocupación. | EP03 | 
| US18 | Historial de alertas e incidencias operativas | **Como** dueño de un restaurante chifa, **quiero** consultar el historial completo de alertas registradas por los sensores y el sistema, **para** identificar patrones de riesgo, tomar medidas correctivas y contar con evidencia de los eventos operativos. | **Escenario 1:** **Dado que** el dueño accede al módulo de incidencias. **Cuando** se carga el historial. **Entonces** se listan todas las alertas con fecha, hora, tipo de alerta, área afectada y estado de revisión (revisada / pendiente). <br> <br> **Escenario 2:** **Dado que** el dueño desea analizar un período o tipo específico. **Cuando** aplica filtros por fecha, tipo de alerta o área. **Entonces** el historial muestra solo los registros que coinciden con el criterio seleccionado. <br> <br> **Escenario 3:** **Dado que** el dueño selecciona una alerta del historial. **Cuando** accede a su detalle. **Entonces** puede ver el valor registrado por el sensor, la nota de seguimiento (si fue marcada como revisada) y la hora exacta del evento. <br> <br> **Escenario 4:** **Dado que** el dueño accede al historial. **Cuando** no existen incidencias en el período seleccionado. **Entonces** el sistema muestra un mensaje informativo indicando que no se encontraron registros para ese criterio. | EP03 |
| US19 | Exportar reporte de monitoreo y alertas | **Como** administrador de un restaurante chifa, **quiero** exportar los datos de monitoreo de temperatura y alertas en formato PDF o CSV, **para** compartirlos con el equipo o usarlos en auditorías internas. | **Escenario 1:** **Dado que** el administrador quiere obtener reportes de monitoreo para compartir. **Cuando** accede a la opción de exportar y selecciona el formato PDF o CSV. **Entonces** se genera el reporte con los datos de temperatura y alertas para su uso. <br> <br> **Escenario 2:** **Dado que** el administrador necesita auditoría de los datos. **Cuando** exporta el reporte en los formatos permitidos. **Entonces** se guarda el reporte en el dispositivo del usuario. | EP03 |
| US20 | Registro y perfil del proveedor | **Como** proveedor de insumos, **quiero** crear un perfil en la plataforma con mis datos, catálogo y condiciones de entrega, **para** que los restaurantes puedan encontrarme y enviarme pedidos directamente. | **Escenario 1:** **Dado que** el proveedor accede al formulario de registro. **Cuando** completa nombre, RUC, categorías de productos y zonas de cobertura. **Entonces** su perfil queda creado y visible para los restaurantes vinculados. <br> <br> **Escenario 2:** **Dado que** el proveedor desea actualizar su información. **Cuando** edita su perfil y guarda los cambios. **Entonces** la información actualizada es visible de inmediato para sus clientes vinculados. <br> <br> **Escenario 3:** **Dado que** el proveedor intenta registrarse. **Cuando** deja campos obligatorios vacíos como el RUC o nombre de empresa. **Entonces** el sistema muestra un mensaje de error indicando los campos requeridos y no crea el perfil. | EP04 |
| US21 | Recepción y gestión de órdenes de compra | **Como** proveedor, **quiero** recibir las órdenes de compra de los restaurantes directamente en mi panel, **para** gestionar cada pedido sin depender de canales informales. | **Escenario 1:** **Dado que** un restaurante envía una orden de compra. **Cuando** el proveedor accede a su panel. **Entonces** la orden aparece listada con fecha, restaurante, insumos y cantidades. <br> **Escenario 2:** **Dado que** el proveedor procesa una orden. **Cuando** actualiza su estado (recibida, en preparación, despachada). **Entonces** el restaurante recibe una notificación automática del cambio. <br> <br> **Escenario 3:** **Dado que** el proveedor intenta actualizar el estado de una orden. **Cuando** intenta retroceder el estado a uno anterior (ej. de "despachada" a "pendiente"). **Entonces** el sistema muestra un mensaje de error indicando que el cambio de estado no es permitido. | EP04 |
| US22 | Vista de demanda proyectada de clientes | **Como** proveedor, **quiero** visualizar la demanda estimada de mis clientes restaurantes para los próximos días, **para** planificar mi producción y distribución con anticipación. | **Escenario 1:** **Dado que** el proveedor accede al panel de demanda. **Cuando** se carga la vista. **Entonces** se muestra una proyección agregada por insumo y por cliente para los próximos días. <br> **Escenario 2:** **Dado que** el proveedor desea analizar un cliente específico. **Cuando** aplica el filtro por restaurante o producto. **Entonces** la proyección se actualiza mostrando solo los datos del criterio seleccionado. <br> <br> **Escenario 3:** **Dado que** el proveedor accede al panel de demanda. **Cuando** ninguno de sus clientes vinculados tiene historial de consumo registrado. **Entonces** el sistema muestra un aviso indicando que no hay datos suficientes para mostrar una proyección. | EP04 |
| US23 | Gestión de catálogo de productos | **Como** proveedor, **quiero** administrar mi catálogo con precios, unidades y disponibilidad, **para** que los restaurantes siempre vean información actualizada al generar sus pedidos. | **Escenario 1:** **Dado que** el proveedor accede a su catálogo. **Cuando** agrega un nuevo producto con nombre, precio, unidad y stock disponible. **Entonces** el producto queda visible para los restaurantes vinculados de inmediato. <br> **Escenario 2:** **Dado que** el proveedor desactiva un producto. **Cuando** realiza el cambio. **Entonces** el producto deja de aparecer disponible para los restaurantes sin eliminar su historial. <br> <br> **Escenario 3:** **Dado que** el proveedor intenta agregar un producto a su catálogo. **Cuando** deja campos obligatorios vacíos como nombre o precio. **Entonces** el sistema muestra un mensaje de error indicando los datos faltantes y no guarda el producto. | EP04 |
